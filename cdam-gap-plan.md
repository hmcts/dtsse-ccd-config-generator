# CDAM attach implementation note

## Status

This is no longer a decentralised runtime SDK gap for durable documents that are newly returned from an
about-to-submit callback with a CDAM `document_hash`.

The SDK now mirrors CCD's central behaviour for this path:

* compare case data before and after the about-to-submit callback
* identify document references that are new in the callback result
* read the callback-returned `document_hash`
* call CDAM `PATCH /cases/documents/attachToCase` with the current case id, case type and jurisdiction
* strip `document_hash` before response/persistence
* fail the submission before commit if CDAM attach fails

Documents already present as event input are deliberately not attached by this SDK step.

## Problem

CDAM upload and CDAM case attachment are separate operations.

CCD can attach documents that are present in the event input before submit is delegated to a decentralised service. The
SDK now handles documents generated by the service during about-to-submit callback execution, provided those documents
are returned in durable case data with the CDAM `document_hash`.

The invariant for decentralised callback emulation should be:

> Do not commit case data containing a new durable document reference until CDAM attach has succeeded for the same case
> id, case type and jurisdiction.

This behaviour matters for ET because document generation is decentralised across multiple services and callback flows.
Generated documents are uploaded in places such as docmosis generation, ET1/ET3 generation, ACAS certificate conversion,
listings, Excel reports, referrals, vetting, notifications and TSE flows. Some of those documents are stored in case
data, while others are returned as `significant_item` or embedded in markup.

Threading `caseId` through every ET document generation path would work, but it is fragile. Each new document generation
path would need to remember to call CDAM attach correctly. The SDK attach step removes that requirement for documents
committed into case data by an about-to-submit callback.

## Scope

In scope:

* Documents newly introduced by an about-to-submit callback response.
* Documents that will be committed into durable case data by the decentralised runtime.
* CDAM attach using the current case id, case type and jurisdiction before the transaction commits.

Out of scope:

* Documents already present in the event input. CCD should continue to attach these before delegation where possible.
* Existing legacy documents already stored without CDAM case metadata.
* Documents generated but not stored in case data, unless the runtime explicitly supports scanning callback response
  fields such as `significant_item`.
* Retrofitting ET-specific document generation services as the long-term solution.
* We MUST NOT attempt to attach existing documents, or docs that are present as event input - *STRICTLY ONLY DOCS UPLOADED BY AN ABOUT TO SUBMIT CALLBACK AND RETURNED IN CASE DATA WITH HASH*

## SDK behaviour

The decentralised runtime centrally detects and attaches new documents introduced by about-to-submit callback
execution.

Flow:

1. Capture the case data before invoking about-to-submit callback emulation.
2. Invoke the service callback.
3. Capture the callback result case data.
4. Scan before and after case data for CCD document fields.
5. Identify document ids that are present after callback execution but were not present before it.
6. For each new document id, read the CDAM hash token from the callback-returned `document_hash`.
7. Call CDAM attach with:
   * `caseId`
   * `caseTypeId`
   * `jurisdictionId`
   * document id
   * document hash token
8. Only commit the case data if all required attaches succeed.

Failure behaviour:

* If attach fails, fail the event submission and do not commit the updated case data.
* Surface the failure as an event submission error, not as a later document access error.
* Log document id, case id, case type and jurisdiction, but do not log hash tokens.

## Document discovery

The scanner walks the JSON case data, matching CCD/CDAM document references at a low level. This is intentional because
services such as ET may use domain-specific document models rather than the SDK's generated document type.

It should identify:

* top-level document fields
* documents inside complex types
* documents inside collections and nested collections
* document fields represented by generated SDK document types

It should compare document identity by document id extracted from `document_url` or `document_binary_url`.

The scanner ignores:

* unchanged document references
* removed document references
* malformed document URLs, except for normal validation/reporting
* documents present only in transient/internal fields that are not committed
* documents that do not have a callback-returned hash token

## Hash token handling

CDAM attach requires a hash token. CDAM generates this token when the document is uploaded or when a hash token is
requested. CCD's central flow expected that token to round-trip through callback case data as `document_hash`.

The SDK does the same: it reads `document_hash` from the callback result and strips it before response/persistence. It
does not invent or recompute the token in production code.

The hash token must never be included in generated CCD definitions or committed case data.

## Permissions

The submitting service needs CDAM `ATTACH` permission for the relevant jurisdiction and case types.

For ET this means the service account used by the callback runtime must be authorised in CDAM service config for:

* jurisdiction `EMPLOYMENT`
* relevant ET case types, or a safe wildcard if accepted by CDAM policy
* permissions including `ATTACH`, and normally `CREATE` and `READ` for generated document flows

The runtime should make permission failures obvious at submit time. A missing `ATTACH` permission should not result in
case data being committed with an inaccessible CDAM document.

## ET implications

ET-specific callback changes are not required for documents that are returned in case data with `document_hash`. Whatever
would previously have gone back over the wire to central CCD now goes to the SDK, and the SDK performs the attach.

The remaining ET considerations are:

* ET has many document generation entry points.
* Some paths use SYA `CaseDocumentService`, some use docmosis `DocumentManagementService`.
* Some docs are event input documents, some are callback-generated documents.
* Some callback-generated docs are stored in case data, some are only returned as markup or `significant_item`.
* callback-generated docs must preserve the CDAM `document_hash` until the SDK strips it

ET would still need a decision for docs that are not committed into case data. If those documents must remain accessible
after CDAM TTL expiry, they need either:

* to be represented in case data, so the SDK can attach them; or
* a specific runtime extension that also scans and attaches documents returned via `significant_item`/markup.

## Migration and legacy documents

This plan does not repair existing documents that already have missing CDAM metadata.

For existing ET documents showing CDAM errors such as `Case Type Id is null`, likely causes are:

* old DM Store uploads
* CDAM uploads performed before attach was introduced
* documents referenced in case data without CDAM case metadata

Those need a separate migration/backfill plan. The migration would need to find document references in case data, map
them to case id/type/jurisdiction, and attach or backfill CDAM metadata using an approved mechanism.

## Test coverage

The e2e legacy JSON test now includes an ACAS-style callback path:

* event input contains an existing document with a deliberately invalid hash token
* about-to-submit callback adds a second document with a CDAM-valid `document_hash`
* SDK attaches only the callback-added document
* hashes are stripped from response and stored case data
* local CDAM records `PATCH /cases/documents/attachToCase` as successful for the callback document

## Open questions

* Should the runtime scan only committed case data, or also `significant_item` and markup fields in callback responses?
* Should attach be best-effort for non-durable callback response fields, or should it fail the event?
* What is the approved approach for backfilling legacy CDAM metadata?
